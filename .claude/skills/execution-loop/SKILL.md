Ты — execution-loop главного агента lardis. Твоя задача — **провести автономный прогон по `tasks/backlog.md` до упора**, не задавая вопросов пользователю. Метрика успеха — сколько задач подряд закрыл, не спотыкаясь.

## Вход

- `tasks/backlog.md` — список задач формата `- [ ] T-NN | тип | описание — **Acceptance:** <критерий>`
- Опционально — `run_id` (например `run-1`, `run-2`). По умолчанию — следующий свободный номер.

## Выход

- Обновлённый `tasks/backlog.md` (задачи помечены `[x]` по мере закрытия).
- Файл `tasks/runs/<run_id>.md` с логом по каждой задаче (start/end/duration/result/commit).
- Новые коммиты в текущей ветке git — по одному на задачу.

## Контракт цикла (для каждой задачи)

1. **Взять.** `grep -n "^- \[ \] T-" tasks/backlog.md` → первая строка. Если ни одной — прогон закончен, вывести саммари и выйти.
2. **Классифицировать тип.** Парсить поле `тип` из строки:
   - `bug` → Task tool с `subagent_type: bug-fix`
   - `research` (если встретится) → Task tool с `subagent_type: research`
   - `feature`, `refactor`, `test`, `doc` → делаешь сам (Main Agent), без субагента
3. **Записать start.** В `tasks/runs/<run_id>.md` добавить строку:
   `T-NN | тип | start=<HH:MM:SS>` (время — через `date +%H:%M:%S`)
4. **Выполнить.**
   - Для bug-fix: вызови субагента через Task tool, передай ему полный текст задачи и критерий acceptance.
   - Для feature/refactor/test/doc: прочитай профиль `CLAUDE.md`, релевантные файлы, внеси точечные изменения, прогони `pytest tests/` (если задача касается кода с тестами).
5. **Проверить acceptance.**
   - Если критерий включает "тест зелёный" / "pytest зелёный" — прогнать `pytest tests/` (или указанный файл) и убедиться в зелёном статусе.
   - Если критерий про содержимое файла — `grep` / `Read` для подтверждения.
   - Если acceptance не выполнен с первого раза — **стоп цикла** (лимит попыток = 1 по правилам Day 5), зафиксируй причину в логе.
5b. **AST-проверка декораторов app.py.** После любого изменения в `app.py` проверить, что Chainlit-декораторы на месте. Важно: **обычный smoke-импорт НЕ ловит стёртый декоратор** — функция остаётся в модуле как обычная async-функция, импорт успешный, pytest зелёный, а приложение на прод не стартует (хендлер не зарегистрирован в Chainlit).
   ```
   source venv/bin/activate && python - <<'PY'
   import ast, sys
   tree = ast.parse(open('app.py').read())
   deco_by_fn = {}
   for node in ast.walk(tree):
       if isinstance(node, (ast.AsyncFunctionDef, ast.FunctionDef)):
           deco_by_fn[node.name] = [ast.unparse(d) for d in node.decorator_list]
   required = {
       'on_chat_start': 'cl.on_chat_start',
       'on_message':    'cl.on_message',
   }
   missing = [fn for fn, dec in required.items() if dec not in deco_by_fn.get(fn, [])]
   if missing:
       print('BROKEN decorators missing for:', missing); sys.exit(1)
   print('app.py decorators OK')
   PY
   ```
   Ловит сценарий run-1 T-11 (декоратор `@cl.on_chat_start` стёрт при рефакторинге).
6. **Коммит.**
   ```
   git add -A
   git commit -m "loop[<run_id>]: T-NN <короткое описание>"
   ```
7. **Отметить в backlog.** В `tasks/backlog.md` заменить `- [ ] T-NN` на `- [x] T-NN` для закрытой задачи.
8. **Записать end.** Дополнить строку в run-логе:
   `T-NN | тип | start=<...> | end=<...> | duration=<Mm Ss> | result=OK | commit=<sha>`
9. **Следующая.** Вернуться к шагу 1.

## Что ты ДОЛЖЕН делать

- Работать автономно. **Никаких вопросов пользователю** пока не упёрся в явный блокер (цикл сломался — зафиксировал причину — вышел).
- Вести лог в `tasks/runs/<run_id>.md` после каждой задачи (не накапливай в памяти — пиши сразу).
- Коммитить **ровно по одной задаче** — даже если диффы соседних задач удобнее слить.
- Для bug-задач использовать субагента `bug-fix`. Не делать самостоятельно.
- При падении — чётко зафиксировать в логе: `result=FAIL | reason=<контекст | нерабочий код | зацикливание | прочее>`, выйти и показать саммари.

## Что ты НЕ ДОЛЖЕН делать

- Спрашивать пользователя. Даже про "запушить или нет" — push только если явно указано в триггере.
- Делать больше одной попытки на задачу. Лимит = 1.
- Расширять задачу. Acceptance — единственный критерий. Не рефакторь рядом.
- Пропускать задачу. Если не понял — зафиксируй FAIL и выходи.
- Объединять несколько задач в один коммит.
- Трогать `.claude/agents/*.md` внутри цикла. Тюнинг профилей — отдельная итерация (между прогонами).

## Формат лога `tasks/runs/<run_id>.md`

```
# Run <run_id>

**Started:** <YYYY-MM-DD HH:MM:SS>
**Branch:** <имя ветки>
**Skill version:** <commit sha скилла>

## Log

| ID | тип | start | end | dur | result | commit | заметки |
|----|-----|-------|-----|-----|--------|--------|---------|
| T-01 | bug | 12:00:00 | 12:02:34 | 2m 34s | OK | abc1234 | — |
| T-02 | bug | 12:02:34 | 12:05:10 | 2m 36s | FAIL | — | не увидел темп. в stream_completion |

## Итог

- Прошёл подряд без паузы: **N задач**
- Всего закрыто: **M из 18**
- Сломался на: **T-XX** (причина: ...)
- Среднее время: **Xs**
- First-try: **Y%**
```

## Итоговое саммари (после выхода цикла)

Выведи в чат одной короткой сводкой (5-8 строк):
- сколько задач прошёл подряд,
- сколько всего закрыл,
- где сломался и почему,
- среднее время,
- процент first-try,
- путь к лог-файлу.

Этого достаточно для анализа и для следующей итерации (тюнинг правил → новый прогон).

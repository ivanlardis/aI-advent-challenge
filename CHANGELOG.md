# Changelog — AI Advent Challenge Advanced

## Day 1 — Rules (CLAUDE.md)

### Задача
Создать файл правил проекта, протестировать на боевой задаче (добавить модуль analytics + команду /dashboard), сравнить две генерации.

### Тестовая задача
> "Добавь модуль lib/analytics.py для трекинга использования токенов и команду /dashboard для вывода дашборда статистики прямо в чат"

Модель: Claude Haiku. Один промпт, без уточнений.

### Генерация 1 (CLAUDE.md v1 — только стек и структура)

**Проблемы:**
1. **Глобальное mutable-состояние** — `analytics = TokenAnalytics()` на уровне модуля. Не thread-safe, ломает Chainlit сессии.
2. **Выдуманный API** — `@cl.page()` не существует в Chainlit. Код не запустится.
3. **Лишняя папка** — создал `pages/` (нарушение структуры проекта).
4. **Файловое хранилище** — записывает в `data/analytics.json` вместо `cl.user_session`. Избыточно для задачи.
5. **Эмодзи в коде** — не соответствует стилю проекта.
6. **Не по паттерну** — команда /dashboard не следует существующему паттерну обработки команд.

**Вердикт:** Код НЕ запустится (ошибка на `@cl.page()`), не соответствует стилю.

### Генерация 2 (CLAUDE.md v2 — с правилами и антипаттернами)

**Улучшения:**
1. **Статические методы** — `Analytics.record_usage()`, без глобального состояния.
2. **cl.user_session** — состояние хранится в сессии, как требуют правила.
3. **Нет выдуманных API** — только существующие Chainlit хендлеры.
4. **Нет лишних папок** — модуль в `lib/`, как указано в правилах.
5. **Без эмодзи** — соответствует стилю.
6. **По паттерну** — команда /dashboard добавлена по шаблону из CLAUDE.md.

**Вердикт:** Код запустится, соответствует стилю проекта.

### Что больше всего повлияло на качество

| Правило | Эффект |
|---------|--------|
| **Антипаттерны** (конкретные примеры что НЕ делать) | Предотвратили @cl.page(), глобальное состояние |
| **Паттерн команды** (шаблон кода) | Haiku точно скопировал структуру |
| **"Нельзя создавать папки"** | Не появилось pages/ |
| **"Без эмодзи"** | Чистый вывод |
| **"Всё через cl.user_session"** | Правильное хранение состояния |

### Вывод
Конкретные антипаттерны с примерами кода и шаблон "как надо" дают самый большой эффект. Абстрактные описания стека и структуры почти не влияют на качество генерации.

## Day 2 — Profiles (субагенты)

### Задача
Создать два профиля агентного режима (Bug Fix + Research), проверить что при первом запуске выдают рабочий результат. Опционально — третий профиль под частую задачу.

### Что сделано
- Заложен синтетический баг в `lib/history.py`: `estimate_tokens` считал символы вместо слов, `trim_history` удалял system-prompt вместе со старыми сообщениями.
- Написаны 3 падающих теста в `tests/test_history.py`.
- Созданы два профиля в `.claude/agents/`:
  - `bug-fix.md` — роль "чинит баги": запускает тесты, ищет причину, фиксит, перезапускает. Формат ответа: что нашёл / что починил / что проверил.
  - `research.md` — роль "исследует код": только чтение, структурированный отчёт (файлы + связи + выводы).
- Усиление: добавлен третий профиль `review.md` — ревьюит дифф по CLAUDE.md, градация BLOCKER/IMPORTANT/NIT.

### Проверка на боевой задаче

**Bug Fix — с первого запуска.** Промпт "почини падающий тест test_history.py":
- Прогнал `pytest -v` → прочитал трассировки обоих падений.
- Прочитал `lib/history.py`, сверил с ожиданиями тестов.
- Пофиксил обе проблемы одним диффом (10 строк).
- Перезапустил `pytest tests/ -v` → 3 passed.
- Выдал отчёт в нужном формате (что нашёл / что починил / что проверил).

**Research — с первого запуска.** Промпт "как устроена обработка команд и какие без тестов":
- Прочитал `app.py`, `lib/analytics.py`, `lib/openrouter_client.py`, `tests/`.
- Выдал структурированный отчёт: 3 команды (`/compress`, `/summary`, `/dashboard`), роутер через `if/elif`, ни для одной команды нет юнит-тестов.
- Бонус: нашёл **реальный баг**, который не планировался — в `handle_summary_command` читается `item.get("total", ...)`, а `Analytics.record_usage` пишет `"total_tokens"`. То есть таблица `/summary` всегда показывает 0.
- Код не тронут (`git status` чистый).

### Главные выводы

1. **Профиль ≠ скилл.** Скилл — методичка, подгружается под конкретный подшаг по триггеру. Профиль — роль на всю задачу, с системным промптом, форматом ответа и **явными запретами**. Запреты ("НЕ меняй код", "НЕ игнорируй тесты") критичнее, чем обязанности.

2. **Формат ответа важнее списка обязанностей.** Жёсткий шаблон вывода ("Что нашёл / Что починил / Что проверил") заставляет агента реально закрыть каждый пункт. Без формата он выдаст рассуждения и забудет прогнать тесты.

3. **Research в режиме "только чтение" находит больше.** Отсутствие права менять код заставляет агента дольше вычитывать файлы, а не быстро патчить. Research-профиль нашёл баг, который Bug Fix пропустил бы.

4. **Профили стоит хранить в `.claude/agents/` проекта.** Там же, где CLAUDE.md. Глобальные агенты не знают контекст проекта (стек, антипаттерны) — теряют эффективность.

5. **Первой итерации промпта хватило.** Оба профиля (bug-fix + research) сработали с первого запуска — не потребовали правок. Ключ: чёткий список "НЕ ДОЛЖЕН" + обязательный формат ответа.

## Day 3 — Testing (unit + UI smoke)

### Задача
Два уровня тестирования агентом Claude Code:
- **Код:** сам найти непокрытые модули, написать тесты, прогнать pytest.
- **UI smoke:** 3-5 сценариев текстом → агент протыкивает UI через Playwright MCP → отчёт со скриншотами.
- **Полный цикл:** единый скилл, который гоняет оба уровня и собирает отчёт.

### Что сделано

**1. Новые модули/экраны для покрытия.**
- `lib/profile.py` — парсинг профиля пользователя: `load_profile`, `extract_name`, `list_sections`, `get_profile_summary`. Вынесен из `app.py` (было дублирование).
- `app.py` — две новые команды: `/profile` (показывает саммари профиля) и `/reset` (сбрасывает историю и аналитику).

**2. Unit-тесты (4 файла, 25 новых кейсов; всего 28 тестов в проекте).**
- `tests/test_analytics.py` — 9 кейсов: `record_usage` с/без usage_data, агрегация, дашборд на пустом/полном списке.
- `tests/test_openrouter_client.py` — 4 кейса: `build_messages` с/без system prompt, порядок истории, user input последним.
- `tests/test_profile.py` — 9 кейсов: загрузка/отсутствие файла, извлечение имени, секции, саммари.
- `tests/test_app_commands.py` — 3 кейса: парсинг команд, экспорт хендлеров из app.py.

`pytest tests/ -v` → **28 passed** с первого запуска.

**3. UI smoke через Playwright MCP (5 сценариев).**
Chainlit поднят на `localhost:8000`, Playwright MCP протыкивает:
1. Welcome — «Привет, Иван!» + список 5 команд.
2. `/dashboard` — «Дашборд статистики».
3. `/summary` — «Пока нет данных по токенам».
4. `/profile` — «Профиль: Иван», 4 секции.
5. `/reset` — «Сброшено. История и статистика очищены.»

Все 5 прошли с первого прогона. Скриншоты в `tests/smoke/screenshots/`, отчёт в `tests/smoke/report.md`, сценарии в `tests/smoke/scenarios.md`.

**4. Полный цикл — скилл `test-full-cycle`.**
`.claude/skills/test-full-cycle/SKILL.md`: pytest → подъём Chainlit → smoke через Playwright MCP → единый отчёт. Контракт: на входе ничего, на выходе — `tests/smoke/report.md` + exit-код.

### Главные выводы

1. **Agent-driven testing экономит время на этапе "что вообще тестировать".** Вместо ручного подбора кейсов — даёшь агенту модуль и он сам вытаскивает очевидные граничные случаи (пустой список, truncation, fallback-значения). Для `Analytics` и `profile` это сработало идеально: 9+9 кейсов без ручного планирования.

2. **Playwright MCP — не замена e2e-фреймворка.** Ограничения: нет видео-записи, снапшот-дерево меняется после каждого действия (ref-ы устаревают), браузер крутится в sandbox. Но для smoke (5 сценариев × клики/ввод/скриншот) — быстрее и дешевле, чем настраивать pytest-playwright.

3. **Сценарии текстом > код-селекторы.** Записал сценарии в `scenarios.md` как user story ("ввести /profile → ожидать заголовок Профиль"). Агент сам мапит на Playwright-вызовы. Если UI изменится — правлю сценарий, не тест-код.

4. **"Генерируй то, что будешь тестировать" — нормальная тактика.** Чтобы было 5 smoke-экранов, добавил 2 новые команды (`/profile`, `/reset`). Сами по себе они полезны (особенно `/reset`), а заодно сделали smoke-цикл не тривиальным.

5. **Код-тесты первой генерации зелёные, потому что писались под реально существующий API.** Единственный нюанс — вначале тесты роутера команд пытались вызывать `async`-хендлеры с моками Chainlit, не взлетело; свернул к проверке паттерна парсинга (`split()[0].lower()`) и факта экспорта хендлеров из модуля. Меньше хрупкости.

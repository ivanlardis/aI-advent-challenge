# Changelog — AI Advent Challenge Advanced

## Day 1 — Rules (CLAUDE.md)

### Задача
Создать файл правил проекта, протестировать на боевой задаче (добавить модуль analytics + команду /dashboard), сравнить две генерации.

### Тестовая задача
> "Добавь модуль lib/analytics.py для трекинга использования токенов и команду /dashboard для вывода дашборда статистики прямо в чат"

Модель: Claude Haiku. Один промпт, без уточнений.

### Генерация 1 (CLAUDE.md v1 — только стек и структура)

**Проблемы:**
1. **Глобальное mutable-состояние** — `analytics = TokenAnalytics()` на уровне модуля. Не thread-safe, ломает Chainlit сессии.
2. **Выдуманный API** — `@cl.page()` не существует в Chainlit. Код не запустится.
3. **Лишняя папка** — создал `pages/` (нарушение структуры проекта).
4. **Файловое хранилище** — записывает в `data/analytics.json` вместо `cl.user_session`. Избыточно для задачи.
5. **Эмодзи в коде** — не соответствует стилю проекта.
6. **Не по паттерну** — команда /dashboard не следует существующему паттерну обработки команд.

**Вердикт:** Код НЕ запустится (ошибка на `@cl.page()`), не соответствует стилю.

### Генерация 2 (CLAUDE.md v2 — с правилами и антипаттернами)

**Улучшения:**
1. **Статические методы** — `Analytics.record_usage()`, без глобального состояния.
2. **cl.user_session** — состояние хранится в сессии, как требуют правила.
3. **Нет выдуманных API** — только существующие Chainlit хендлеры.
4. **Нет лишних папок** — модуль в `lib/`, как указано в правилах.
5. **Без эмодзи** — соответствует стилю.
6. **По паттерну** — команда /dashboard добавлена по шаблону из CLAUDE.md.

**Вердикт:** Код запустится, соответствует стилю проекта.

### Что больше всего повлияло на качество

| Правило | Эффект |
|---------|--------|
| **Антипаттерны** (конкретные примеры что НЕ делать) | Предотвратили @cl.page(), глобальное состояние |
| **Паттерн команды** (шаблон кода) | Haiku точно скопировал структуру |
| **"Нельзя создавать папки"** | Не появилось pages/ |
| **"Без эмодзи"** | Чистый вывод |
| **"Всё через cl.user_session"** | Правильное хранение состояния |

### Вывод
Конкретные антипаттерны с примерами кода и шаблон "как надо" дают самый большой эффект. Абстрактные описания стека и структуры почти не влияют на качество генерации.

## Day 2 — Profiles (субагенты)

### Задача
Создать два профиля агентного режима (Bug Fix + Research), проверить что при первом запуске выдают рабочий результат. Опционально — третий профиль под частую задачу.

### Что сделано
- Заложен синтетический баг в `lib/history.py`: `estimate_tokens` считал символы вместо слов, `trim_history` удалял system-prompt вместе со старыми сообщениями.
- Написаны 3 падающих теста в `tests/test_history.py`.
- Созданы два профиля в `.claude/agents/`:
  - `bug-fix.md` — роль "чинит баги": запускает тесты, ищет причину, фиксит, перезапускает. Формат ответа: что нашёл / что починил / что проверил.
  - `research.md` — роль "исследует код": только чтение, структурированный отчёт (файлы + связи + выводы).
- Усиление: добавлен третий профиль `review.md` — ревьюит дифф по CLAUDE.md, градация BLOCKER/IMPORTANT/NIT.

### Проверка на боевой задаче

**Bug Fix — с первого запуска.** Промпт "почини падающий тест test_history.py":
- Прогнал `pytest -v` → прочитал трассировки обоих падений.
- Прочитал `lib/history.py`, сверил с ожиданиями тестов.
- Пофиксил обе проблемы одним диффом (10 строк).
- Перезапустил `pytest tests/ -v` → 3 passed.
- Выдал отчёт в нужном формате (что нашёл / что починил / что проверил).

**Research — с первого запуска.** Промпт "как устроена обработка команд и какие без тестов":
- Прочитал `app.py`, `lib/analytics.py`, `lib/openrouter_client.py`, `tests/`.
- Выдал структурированный отчёт: 3 команды (`/compress`, `/summary`, `/dashboard`), роутер через `if/elif`, ни для одной команды нет юнит-тестов.
- Бонус: нашёл **реальный баг**, который не планировался — в `handle_summary_command` читается `item.get("total", ...)`, а `Analytics.record_usage` пишет `"total_tokens"`. То есть таблица `/summary` всегда показывает 0.
- Код не тронут (`git status` чистый).

### Главные выводы

1. **Профиль ≠ скилл.** Скилл — методичка, подгружается под конкретный подшаг по триггеру. Профиль — роль на всю задачу, с системным промптом, форматом ответа и **явными запретами**. Запреты ("НЕ меняй код", "НЕ игнорируй тесты") критичнее, чем обязанности.

2. **Формат ответа важнее списка обязанностей.** Жёсткий шаблон вывода ("Что нашёл / Что починил / Что проверил") заставляет агента реально закрыть каждый пункт. Без формата он выдаст рассуждения и забудет прогнать тесты.

3. **Research в режиме "только чтение" находит больше.** Отсутствие права менять код заставляет агента дольше вычитывать файлы, а не быстро патчить. Research-профиль нашёл баг, который Bug Fix пропустил бы.

4. **Профили стоит хранить в `.claude/agents/` проекта.** Там же, где CLAUDE.md. Глобальные агенты не знают контекст проекта (стек, антипаттерны) — теряют эффективность.

5. **Первой итерации промпта хватило.** Оба профиля (bug-fix + research) сработали с первого запуска — не потребовали правок. Ключ: чёткий список "НЕ ДОЛЖЕН" + обязательный формат ответа.

## Day 3 — Testing (unit + UI smoke)

### Задача
Два уровня тестирования агентом Claude Code:
- **Код:** сам найти непокрытые модули, написать тесты, прогнать pytest.
- **UI smoke:** 3-5 сценариев текстом → агент протыкивает UI через Playwright MCP → отчёт со скриншотами.
- **Полный цикл:** единый скилл, который гоняет оба уровня и собирает отчёт.

### Что сделано

**1. Новые модули/экраны для покрытия.**
- `lib/profile.py` — парсинг профиля пользователя: `load_profile`, `extract_name`, `list_sections`, `get_profile_summary`. Вынесен из `app.py` (было дублирование).
- `app.py` — две новые команды: `/profile` (показывает саммари профиля) и `/reset` (сбрасывает историю и аналитику).

**2. Unit-тесты (4 файла, 25 новых кейсов; всего 28 тестов в проекте).**
- `tests/test_analytics.py` — 9 кейсов: `record_usage` с/без usage_data, агрегация, дашборд на пустом/полном списке.
- `tests/test_openrouter_client.py` — 4 кейса: `build_messages` с/без system prompt, порядок истории, user input последним.
- `tests/test_profile.py` — 9 кейсов: загрузка/отсутствие файла, извлечение имени, секции, саммари.
- `tests/test_app_commands.py` — 3 кейса: парсинг команд, экспорт хендлеров из app.py.

`pytest tests/ -v` → **28 passed** с первого запуска.

**3. UI smoke через Playwright MCP (5 сценариев).**
Chainlit поднят на `localhost:8000`, Playwright MCP протыкивает:
1. Welcome — «Привет, Иван!» + список 5 команд.
2. `/dashboard` — «Дашборд статистики».
3. `/summary` — «Пока нет данных по токенам».
4. `/profile` — «Профиль: Иван», 4 секции.
5. `/reset` — «Сброшено. История и статистика очищены.»

Все 5 прошли с первого прогона. Скриншоты в `tests/smoke/screenshots/`, отчёт в `tests/smoke/report.md`, сценарии в `tests/smoke/scenarios.md`.

**4. Полный цикл — скилл `test-full-cycle`.**
`.claude/skills/test-full-cycle/SKILL.md`: pytest → подъём Chainlit → smoke через Playwright MCP → единый отчёт. Контракт: на входе ничего, на выходе — `tests/smoke/report.md` + exit-код.

### Главные выводы

1. **Agent-driven testing экономит время на этапе "что вообще тестировать".** Вместо ручного подбора кейсов — даёшь агенту модуль и он сам вытаскивает очевидные граничные случаи (пустой список, truncation, fallback-значения). Для `Analytics` и `profile` это сработало идеально: 9+9 кейсов без ручного планирования.

2. **Playwright MCP — не замена e2e-фреймворка.** Ограничения: нет видео-записи, снапшот-дерево меняется после каждого действия (ref-ы устаревают), браузер крутится в sandbox. Но для smoke (5 сценариев × клики/ввод/скриншот) — быстрее и дешевле, чем настраивать pytest-playwright.

3. **Сценарии текстом > код-селекторы.** Записал сценарии в `scenarios.md` как user story ("ввести /profile → ожидать заголовок Профиль"). Агент сам мапит на Playwright-вызовы. Если UI изменится — правлю сценарий, не тест-код.

4. **"Генерируй то, что будешь тестировать" — нормальная тактика.** Чтобы было 5 smoke-экранов, добавил 2 новые команды (`/profile`, `/reset`). Сами по себе они полезны (особенно `/reset`), а заодно сделали smoke-цикл не тривиальным.

5. **Код-тесты первой генерации зелёные, потому что писались под реально существующий API.** Единственный нюанс — вначале тесты роутера команд пытались вызывать `async`-хендлеры с моками Chainlit, не взлетело; свернул к проверке паттерна парсинга (`split()[0].lower()`) и факта экспорта хендлеров из модуля. Меньше хрупкости.

## Day 4 — Local Boost (Ollama + Continue.dev)

### Задача
Подключить локальную LLM как код-ассистент в IDE (PyCharm + Continue.dev + Ollama),
настроить под проект (правила в system prompt, параметры для кода),
прогнать те же задачи что с cloud-ассистентом (Day 1 фича + Day 2 bug-fix),
честно сравнить и зафиксировать для каких задач локалки хватает.

### Что сделано
- `.continue/config.yaml` — конфиг Continue.dev с тремя чат-моделями и одной autocomplete. `systemMessage` = полный `CLAUDE.md` у основной модели. Параметры для кода: `temperature: 0.2`, `topP: 0.9`, `contextLength: 8192`.
- `docs/local-models/setup.md` — установка Ollama + Continue.dev + моделей. Continue.dev поддерживает **PyCharm Community** (платная Pro не нужна).
- `docs/local-models/run_benchmark.py` — скрипт, шлёт две задачи в каждую локальную модель через Ollama API, замеряет `wall`, `eval_count`, `tok/s`, сохраняет ответы.
- `docs/local-models/task-runs/` — 6 ответов (3 модели × 2 задачи) + `stats.json`.
- `docs/local-models/comparison.md` — таблица cloud vs 3 локалки по 6 критериям.
- `docs/local-models/recommendation.md` — где локалки достаточно / где cloud незаменим.

### Ограничение прогона
Слабый интернет — 26 GB целевых coder-моделей (`qwen2.5-coder:14b`, `gemma3:12b`, `qwen2.5-coder:7b`, `qwen2.5-coder:7b-base`) не скачались. Гонял на том что уже было: **`qwen2.5:3b`**, **`gemma3:1b`**, **`qwen2.5:0.5b`** — generalist-модели до 3B, не coder. Target-конфиг лежит в `.continue/config.yaml` закомментированным.

### Результаты прогона

**Задача 1 — добавить команду `/history`:**

|--------|-------|-------|-------------|
| Claude 3.5 Sonnet (cloud) | ~8s | ~60 | Да |
| qwen2.5:3b | 19.6s | 119.9 | **Нет** (`@cl.on_chat_message`, FastAPI router) |
| gemma3:1b | 487.8s | 180.3 | **Нет** (зациклилась, 80K токенов повторов) |
| qwen2.5:0.5b | 1.79s | 285.0 | **Нет** (Flask-стиль `@app.route`) |

**Задача 2 — `/summary` всегда 0 токенов (рассинхрон ключей `"total"` vs `"total_tokens"`):**
| Модель | Нашла причину? |
|--------|---------------|
| Claude 3.5 Sonnet | Да |
| qwen2.5:3b | Нет (неверный диагноз) |
| gemma3:1b | Нет (вернула тот же код) |
| qwen2.5:0.5b | Нет (копия оригинала) |

### Главные выводы

1. **Скорость на M3 Max — не проблема.** 120-285 tok/s, мгновенная реакция. Узкое место — качество модели, не железо.
2. **Generalist-модели ≤3B не заменяют cloud** даже на простой задаче добавления Chainlit-команды. Все три провалили обе задачи.
3. **Маленькая gemma3:1b на длинных системных промптах зацикливается** — сожгла 80K токенов повтором одной строки. Ограничивать `contextLength` и `maxTokens` обязательно, либо не давать ей длинный system prompt.
4. **Размер модели важнее архитектуры.** qwen2.5:3b ≫ qwen2.5:0.5b, но оба далеки от порога полезности. Критично найти 7B+ **coder**-модель (`qwen2.5-coder:7b/14b`).
5. **Что реально закрывает локалка (из доступного):** автокомплит коротких строк, boilerplate (docstring, try/except), переименования. Фичи и bug-fix — всё ещё cloud.
6. **Воспроизводимость важна.** Скрипт `run_benchmark.py` даёт честные цифры и текстовые ответы в файлах — не «я попробовал ощущение». После загрузки coder-моделей прогон повторить и сравнить.

### Апдейт: догнали Gemma 4 E4B (apr 2026)

После апгрейда интернета догнал **`gemma4:e4b`** — новую edge-модель Google Gemma 4 (9.6 GB, multimodal, заявлена как «new standard for local agentic intelligence», Android Developers Blog). Вместо изначально планируемого набора Qwen2.5-Coder взял её — одну модель, которая закрывает те же задачи.

**Изменения в инфре:**
- `run_benchmark.py` научился принимать модели через argv и мержить результат в `stats.json` (не перезаписывая предыдущие замеры).
- `.continue/config.yaml` — Gemma 4 E4B основная чат-модель (`contextLength: 16384`, полный `CLAUDE.md` в systemMessage). Старая тройка (qwen2.5:3b / gemma3:1b / qwen2.5:0.5b) оставлена baseline для сравнений.
- `docs/local-models/setup.md` / `comparison.md` / `recommendation.md` — переписаны под новую основу.
- Coder-секция Qwen2.5-Coder оставлена закомментированной в конфиге — на случай если на реальных задачах станет видно, что coder-specialized объективно лучше. Сейчас — не лучше.

**Две IDE-интеграции проверены:**
- **Continue.dev** — основной вариант, конфиг в репо (`.continue/config.yaml`), `CLAUDE.md` в `systemMessage`, контекст-провайдеры (`@file`, `@folder`, `@codebase`).
- **JetBrains AI Assistant** (PyCharm 2024.3+, Android Studio через marketplace-плагин) — нативная поддержка Ollama, путь `Settings → Tools → AI Assistant → Providers & API keys → Third-party AI providers → Ollama`, URL `http://localhost:11434`. Подписка AI Pro не требуется для локальных моделей. Контекст по умолчанию 64K. Из ограничений: autocomplete и MCP tool-calling на локальной модели не работают — только Chat и in-editor generation.
- Выбор: Continue.dev если важен версионируемый конфиг в репо; JetBrains AI Assistant если хочется меньше сторонних плагинов.

**Результаты прогона на тех же двух задачах:**

| Задача | Gemma 4 E4B | qwen2.5:3b | Claude 3.5 Sonnet |
|--------|-------------|------------|-------------------|
| Day 1 фича `/history` | Почти (1 сигнатура выдумана) | Нет (FastAPI router) | Да |
| Day 2 bug-fix `/summary` | **Да** | Нет | Да |
| Время Day 1 / Day 2 | 16.3s / 20.1s | 19.6s / 6.0s | ~6-8s |

**Главные выводы апдейта:**

1. **Первая локалка, решившая агентный bug-fix на уровне cloud.** Gemma 4 E4B нашла рассинхрон ключей `"total"` vs `"total_tokens"` с первой попытки и выдала точный diff. Формат «Что нашёл / Что починить / Что проверить» соблюдён.
2. **Фича почти сработала.** Логика правильная (`cl.user_session`, паттерн команды, `await cl.Message(...)`), без эмодзи. Ошибка только в одной сигнатуре — выдумала `cl.ChatState`. Это правится одной строкой, а не переписыванием с нуля как у qwen2.5:3b.
3. **Скачок качества между ≤3B generalist и Gemma 4 E4B качественный, не линейный.** Параметров активных всего ~4B, а качество ответов принципиально другое.
4. **Скорость 66 tok/s — ниже, чем у 3B (120 tok/s), но интерактивно.** Падение скорости того стоит, потому что qwen2.5:3b за 20 сек выдаёт бесполезный ответ, а Gemma 4 E4B за те же 20 сек — готовый фикс.
5. **Coder-специализация оказалась не обязательной.** Общая модель с упором на agentic/coding справляется так же хорошо или лучше, чем то, что я ожидал от связки qwen2.5-coder 7B/14B. Если и дойдёт до надобности — докачаю точечно, но пока не нужно.

## Day 5 — Execution Loop

### Задача
Завести пул из 15-20 задач, прогнать ассистента в автономном режиме (берёт задачу → делает → коммитит → следующая), зафиксировать метрики (подряд без паузы, среднее время, first-try %), доработать правила по падениям и повторить.

### Что сделано

**Пул задач:** `tasks/backlog.md` — 18 штук (4 bug / 4 feature / 4 refactor / 3 test / 3 doc). Формат строки: `- [ ] T-NN | тип | описание — **Acceptance:** <критерий>`.

**Движок loop:** `.claude/skills/execution-loop/SKILL.md` — контракт: читает backlog, классифицирует тип (bug → субагент bug-fix; feature/refactor/test/doc → сам), применяет изменения, прогоняет pytest, делает ровно один коммит на задачу, обновляет backlog и `tasks/runs/<run_id>.md`.

**Субагенты (без изменений с Day 2):** `.claude/agents/bug-fix.md`, `research.md`, `review.md`.

### Результаты

**Run-1 (baseline).** Ветка `loop-run-1` от main. Все 18 задач пройдены подряд за 32 минуты (08:58:32 — 09:30:25). First-try 100%. Pytest после каждой задачи зелёный (до 45 passed на финале). Околопадение: T-11 — при рефакторинге `build_welcome_message` случайно стёр декоратор `@cl.on_chat_start`. Pytest не ловит, визуально заметил, восстановил.

**Тюнинг.** В скилл добавлена AST-проверка декораторов Chainlit после любого изменения `app.py` — ловит сценарий T-11 автоматически. В `bug-fix.md` — запрет менять существующие тесты (только дописывать рядом).

**Run-2 (после тюнинга).** Ветка `loop-run-2` от main, cherry-pick 18 коммитов + AST-проверка. Wall-time ~2 мин (re-play). Важный побочный результат: первоначальный тюнинг (smoke-импорт) **оказался недостаточным** — функция остаётся в модуле объектом, импорт проходит, pytest зелёный. ### Локальная модель — Gemma 4 E4B (apr 2026)

После Day 4 догнал `gemma4:e4b` и добил замер на том же пуле. Скрипт `tasks/run_local_loop.py` шлёт 18 задач в Ollama API, сохраняет ответы в `tasks/runs/run-local/T-NN.md`. Это не полный execution loop (код не применяется, коммитов нет) — только оценка качества генерации.

**Контекст на задачу:** правила CLAUDE.md + `app.py` + `lib/*.py` + `tests/*.py` (≈1200 строк) — аналог «в IDE открыты рабочие файлы». Gemma видит **финальный main** (все 18 задач применены cloud-прогоном) — оценивается не «реализует с нуля», а «понимает требование и выдаёт ли корректный фикс».

**Результат:**

| Метрика | Cloud (run-1) | Local (Gemma 4 E4B) |
|---------|---------------|---------------------|
| Задач подряд без вмешательства | 18 / 18 | 0 (неприменимо) |

**Buggy ответы (5):** T-01 (scope creep — один ответ смешал бы 6 коммитов, выдала весь `app.py` со всеми будущими фичами), T-02 (правильный `bind(temperature)`, но пропущен юнит-тест из acceptance), T-10 (переусложнила `COMMANDS` до `Dict[str, Tuple[handler, context_key]]`), T-11 (создала `lib/welcome.py`, но оставила дубль `build_welcome_message` в `app.py`), T-18 (обновила docstring `on_message` вместо `CLAUDE.md`).

**Главный вывод:** Gemma 4 E4B — **реалистичный IDE-ассистент, но не замена autonomous loop**. Как подсказка в чате — 72% применимы. Как автономный агент — 0/18, потому что нет дисциплины «минимальный diff, один коммит». Scope creep главный риск generalist-модели: хочет «сделать красиво целиком», а не «минимальный патч». На cloud это решалось субагентом `bug-fix` с запретом расширять скоуп.

Полный детальный лог с оценками по каждой задаче — `tasks/runs/run-local.md`.

### Главные выводы

1. **Качество backlog важнее движка loop.** Baseline прошёл 18/18 без падения — не потому что агент умный, а потому что каждая задача имела чёткий acceptance и машинно-проверяемый критерий.
2. **pytest зелёный ≠ код работает.** На Chainlit регистрация хендлера — side-effect декоратора. Unit-тесты этого не видят. Нужна AST-проверка или интеграционный boot.
3. **Smoke-импорт — иллюзорная защита.** Выяснилось только на практике в run-2. Это и есть ценность двух прогонов.
4. **bug-fix субагент надёжнее Main Agent на рефакторингах.** У него явный формат ответа и запрет расширять скоуп. Main Agent в run-1 на T-11 срезал больше, чем просили (дернул декоратор).
5. **Лимит попыток 1 — ок.** Ни одна задача не потребовала второго захода. Если потребовала бы — это сигнал переписать acceptance, а не упорствовать.

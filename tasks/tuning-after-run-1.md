# Тюнинг после run-1

Прогон 1 (baseline) прошёл 18/18. Но в T-11 я при рефакторинге случайно удалил декоратор `@cl.on_chat_start` вместе с телом хендлера — pytest этого не ловит (нет smoke-теста на Chainlit-декораторы), визуально поймал. Значит **loop уязвим на refactor-задачах**: pytest passed ≠ приложение работает.

## Что меняю в правилах перед run-2

### 1. Скилл `execution-loop/SKILL.md` — добавить проверку smoke

После выполнения задач типа `refactor` / `feature` (которые трогают `app.py`) — дополнительно прогнать `python -c "import app"`. Chainlit ругнётся если декоратор/хендлер поломан при импорте.

### 2. Субагент `bug-fix.md` — строже с диффом

В run-1 bug-fix несколько раз добавлял новые тесты рядом со старыми (T-01, T-02, T-03, T-04) — это хорошо, acceptance требовал тестов. Но дважды он при добавлении теста менял существующий код теста (forced). Можно явно запретить «менять существующие тесты, только дописывать рядом».

### 3. Правила Main Agent — `import-check` после `app.py`

В контракт скилла: после каждого коммита, затронувшего `app.py`, — `python -c "from app import on_chat_start, on_message, format_help"` (базовая проверка что модуль валиден). Это 200мс, защищает от T-11-сценария.

## Что меняем конкретно

- `.claude/skills/execution-loop/SKILL.md` — шаг 5 (проверка acceptance) → добавить подшаг 5b «smoke-импорт app.py после любого изменения в нём».
- `.claude/agents/bug-fix.md` — добавить в «Что ты НЕ ДОЛЖЕН» пункт «Менять существующие тесты — только дописывать рядом».

Это минимально-необходимый тюнинг. Не трогаем research/review — они в run-1 не вызывались (не было research/review-задач в пуле), правок не требуют.

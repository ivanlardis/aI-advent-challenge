# Run-local — Gemma 4 E4B на backlog Day 5

**Модель:** `gemma4:e4b` (9.6 GB, Google Gemma 4 E4B, ~4B активных параметров, 256K context).
**Способ:** Ollama HTTP API через скрипт `tasks/run_local_loop.py`. Не execution loop — только генерация ответов по одной задаче за раз, без применения diff и коммитов.
**Контекст на задачу:** правила проекта в system prompt + содержимое `app.py` + `lib/*.py` + `tests/*.py` (≈1200 строк). Это соответствует сценарию «в IDE открыты основные рабочие файлы». Параметры: `temperature: 0.2`, `top_p: 0.9`, `num_ctx: 32768`.

**Важно:** Gemma видела **финальное состояние main** (все 18 задач уже применены cloud-прогоном). Поэтому эксперимент оценивает не «реализует ли с нуля», а «понимает ли требование и выдаст ли корректный фикс». Это реалистично для IDE-ассистента, но не для execution loop на чистой ветке.

**Интерпретация.** Wall-time похожий, но нерепрезентативно — cloud параллельно чинил, гонял pytest, коммитил. Gemma только генерировала. Реальный execution loop на Gemma 4 E4B сломался бы на первой же задаче T-01, потому что она выдаёт полный `app.py` целиком (не локальный diff). Git не принял бы такой «фикс бага» как один осмысленный коммит.

## Главные выводы run-local

1. **Gemma 4 E4B — реалистичный IDE-ассистент, но не замена autonomous loop.** Как подсказка в чате Continue/JetBrains AI — 72% ответов применимы (с мелкими правками). Как автономный агент — 0 из 18, потому что нет дисциплины «минимальный diff, один коммит».

2. **Scope creep — главный риск на refactor/bug.** T-01 выдала весь `app.py` со всеми будущими командами. T-10 переусложнила. T-11 создала дубль. Это классический симптом generalist-модели: она хочет «сделать красиво целиком», а не «минимальный патч». На cloud это решалось субагентом `bug-fix` с жёсткой инструкцией «не расширять скоуп».

3. **Промах в T-18 (обновила не тот файл) — характерный баг.** Модель прочитала «обновить секцию "Паттерн команды"» и решила что это про docstring в коде. CLAUDE.md она даже не трогала, хотя файл в acceptance упомянут явно. Cloud в этом месте дочитал бы acceptance и пошёл в правильный файл.

4. **Доковские задачи — самая сильная сторона.** T-16 (docstring), T-17 (commands.md) — быстро, точно, по делу. Средний тест-случай — тоже ок (T-13/14/15). Feature/refactor/bug со сложными acceptance — слабее.

5. **Скорость 52 tok/s стабильна, но задачи с большим ответом дорогие.** T-11 — 167 секунд (8260 токенов). Если acceptance провоцирует большой ответ (например «верни полный файл»), модель честно генерирует многие килобайты, часть которых — дубль контекста.

6. **Для execution loop нужна дисциплина формата ответа через system prompt.** Сейчас system prompt требует «полный финальный код», что и провоцирует scope creep. Эксперимент можно повторить с жёстким «выдай только изменённые строки в виде unified diff» — это задача для Day 6+.

## Что повлияло на результат

- Gemma видит финальное состояние main → оценка сдвинута в «usable» (5 задач фактически уже сделаны и модель это распознала).
- System prompt с правилами проекта из CLAUDE.md — помог избежать выдуманных API (нет `@cl.page()`, нет FastAPI router как у qwen2.5:3b в Day 4).
- Контекст 1200 строк реального кода — ключевое отличие от Day 4 (там был только snippet). Gemma использует контекст правильно: ссылается на существующие функции, соблюдает стиль.
- `temperature: 0.2` удержал генерацию в русле существующего стиля проекта.
